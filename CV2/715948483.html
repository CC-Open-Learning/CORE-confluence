User Centered Design

Most of the products developed at the VARLab are not aimed at computer savvy users. This means that the people developing them should not rely on complex interface models that they take for granted.

Developers might not be familiar with the audience’s lingo and using it incorrectly will take the user out of the experience. Make sure to always validate the design with the Content Experts.

When designing your interface, keep the goal of the interaction in mind. What is the concept that you are trying to get across? How does the user show us that they understand the concept? Are we leading the user to the developer’s intended flow or are we following the process that they would follow in the real world? The objective might be getting as close to digital twinning as possible or it might be an abstract representation of the actual interaction. Make sure you know which one is the appropriate one for your case and be consistent (i.e. try not to jump from one approach to the other constantly).

Consistency

UI is at its best when it’s invisible. This does not mean that users don’t see it or that it is out of the way, it means that users are not constantly thinking “this is a User Interface”. One of the main things that make this thought pop up is having clashing styles and elements that constantly change their shape and/or colour even in the same user flow.

To avoid taking people out of the situation, designers and developers must aim towards a consistent look and feel for their UI. This is achieved by following design patters (always use the same padding, margins, button positions, and so on) and following already established interfacing standards (the “close” button should be at the top right if it is an [X], if you have an OK button you usually have a Cancel one next to it or if there is no canceling, your OK button will be centered). If for one type of interaction you have your call to action on a button that says “Next”, don’t swap that for a “Continue” in the following step.

Avoid making your user constantly relearn the way they interact with the UI. Using a standardized library and reusable components/prefabs should help changes to the base element to be reflected across your application. Always check that these changes don’t create conflicts across the different implementations and remember that these elements might be in use by other teams so making changes will impact the way they are using them as well.

These are principles that everyone should always refer to when designing an interface. Fitts' Law

Perhaps one of the most commonly referred to principles in UX, Fitts' Law gets us to think about design aimed towards optimization in the user interface.

The simple interpretation is that when designing, we should be conscious of the distance between the user’s position and their interaction’s goal. For example: if a user clicks on a button that will unfold a menu, that menu should be next to the place where the user clicked and not across the screen:

Not following this principle leaves more room for human error, either by risking the user missing out the result of their interaction, misclicking when going towards their intended target or simply by extending the required time to interact with the system, making the interaction less efficient than it could be.

This does not apply only to cursor/touch based interactions. Take this into account when designing your feedback system. If your feedback about a task being completed is far away from the place where the user completed the task, they might miss it. If they failed the task, there’s a chance of them never finding out that they failed and thinking the interaction is broken.
