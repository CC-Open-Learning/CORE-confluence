CORE Framework : Handling the Learner Session

Created by David Inglis, last modified on Apr 09, 2025

Summary

This document defines the Learner Session for a Unity DLX and describes the sequence of events for setting up a session upon launch of a DLX.

The Learner Session is the data describing a user and their actions in the DLX during an instance of the DLX, which is typically loaded from an LMS like eConestoga.

Systems

The main systems that make up the learner session are:

  • Session Provider, handling unique identifying data for a learner

    • SCORM is the main session provider used by existing DLX projects in eConestoga.

    • LTI is an alternative identification system for modern LMSâ€™s which is currently being developed and will replace or work alongside SCORM in the future

  • Session Handler, acting as a communicator between the Session Provider and various systems wanting to use learner data

  • Cloud Save, which stores and retrieves data persisting across multiple sessions, and

  • Analytics, providing event tracking and more intricate features such as version control

The learner session is complemented by the VELCRO UI system, which presents relevant session information to the learner through the introductory splash screen, informational toast messages and dialogs, and a settings menu.

Session Providers

The following Session Providers are described in their order of priority, as defined in the DLX Template project.

LTI

SCORM

Local

  • LearningToolsSessionProvider

    • uses VARLab.Authentication.LTI to attempt a web-based LTI connection

  • ScormSessionProvider

    • uses VARLab.SCORM to attempt a web-based SCORM connection

  • DebugSessionProvider

    • only available in Editor or development builds

    • can be enabled in code or serialized fields in editor

    • optionally uses a static ID for debugging purposes

  • FallbackSessionProvider

    • guaranteed to activate if any of the previous session providers fail to initialize.

    • provides unique ID per-device which is dependent on UnityEngine.PlayerPrefs (which uses browser cookies on web)
