Debugging

This page is intended to give a quick overview of debugging in UI Toolkit. To read more about UI Toolkit Debugging, visit Unity’s documentation. I also highly encourage a quick skim through the [OUTDATED] Common Pitfalls page.

Debugging Examples

UI Builder Debugging

The runtime debugger can be accessed from the UI Builder in Unity 6+, but if a piece of UI doesn’t behave as expected in the UI builder, it is usually easier to find the issue without it. The most common issues in the UI Builder are inline styles overriding classes, margin/padding and/or styling not applying due to an incorrect theme being previewed.

  1. Ensure either Dark.tss, Light.tss, or the project’s [Name]Dark.tss, and [Name]Light.tss are being used.

All USS variables holding colours, sprites, etc. will not display correctly if the theme file being previewed is set to Unity Default Runtime Theme.

  1. Skim through elements to check for inline styles that could potentially be causing the issue. Inline styles are denoted by a white bar.

  1. Select the “preview” button in the top right of the viewport to help debug hover/click transitions quicker outside of paly mode.

  2. Try to group up where margin and padding are placed on as few elements as possible. This is not required, and not always an option, but it is much easier to debug and match UI to Figma designs if only a handful of elements have margin/padding as opposed to 8px on every VisualElement, etc.

  3. Use the temporary USS class below to help see where each element’s bounds are. This can be especially useful to outline invisible wrapper containers that exist to move child elements around such as the #Header, #Content, and #Footer elements in the screenshot below.

/* DEBUG. Remove this when done */
* {
    border-width: 1px;
    border-color: red;
}

Runtime Debugging

The runtime debugger is similar to your browser’s developer tools by pressing F12 and switching to the “Elements” tab. It will display the UXML hierarchy of every UI document in the scene, the box model, and all applied USS of the selected element.

To access the runtime debugger, Window > UI Toolkit > Debugger

If a specific element or piece of UI is behaving differently from expected during runtime, open the debugger, and select the element to view all applied USS. The properties of elements can also be changed here at runtime using the debugger to test changes. All changes will be discarded when play mode is ended.

In the example below:

  • Red is the hierarchy of every piece of UI active in the scene. This is the demo scene for toasts in VELCRO which has a toolbar, simple toast and complex toast.

  • Orange displays the box model of the currently selected element which displays the size in pixels of the margin, border, padding, and element itself. The currently selected element is the Label #TODO

  • Yellow displays the list of active USS classes, and the outcome of the classes' properties.
