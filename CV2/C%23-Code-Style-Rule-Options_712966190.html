Overview

C# Code Style Rule Options will enforce the developer to follow naming conventions and the style guide shown at https://varlab-dev.atlassian.net/l/cp/He0QRqCr. This will enforce the general style, but will not take quality naming into consideration. Ideally the IDE (Visual Studio or VS Code) will create warnings and/or errors for improper conventions.

To achieve this, the developerâ€™s IDE should use the provided Editor Config file, similar to this guide: https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/code-style-rule-optionsand https://learn.microsoft.com/en-us/dotnet/fundamentals/code-analysis/overview?tabs=net-7#code-style-analysis. Program build can be disabled, but requires additional steps by the developer to configure their machine. Errors may be able to be read from a log file, which can remove the need to halt building.

Comments are not being checked, and the reviewer must still check this before accepting the pull request.

Linting C# Code

To lint C# code for naming and style conventions, we need to build a .csproj file that has one or more .cs files. You can use the docker image you created in Building a Unity Project in an Ubuntu Container to check C# files for naming. The C# linting will be done through the use of an .editorconfig, .csproj file, and the .NET MSBuild Tools.

  1. Download the https://bitbucket.org/VARLab/pull-request-automation-tools/src/main/Linter/.editorconfig from the repository.

  2. Download the https://bitbucket.org/VARLab/pull-request-automation-tools/src/main/Linter/AutomatedProject.csproj from the repository. This ensures that when building the project that the .editorconfig file is considered while building.

  3. Store both of these files, along with your .cs files, inside of a folder, named AutomatedProject.

  4. Run the docker image, ensuring that the AutomatedProject folder you created is shared with the image.
