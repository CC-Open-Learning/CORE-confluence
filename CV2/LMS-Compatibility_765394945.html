SCORM Standard

Overview

The Sharable Content Object Reference Model (SCORM) is a set of technical standards and specifications that define how e-learning should be packaged, delivered, and tracked. The SCORM standard provides a framework for communication between e-learning content and the Learning Management System (LMS). SCORM defines how to create “sharable content objects” (SCOs) that can be reused in different systems and specifies how the LMS should launch and the SCOs.

Communication Structure between LMS and Unity

In the interest of having the SCORM standard create a connection between the LMS (eConestoga) and the DLX projects in Unity, the setup follows an established structure for easy integration.

The SCORM API Structure is a communication that consists of several components, involving the web application (eConestoga), the scorm.js library, and several Unity scripts, including the ScormAPIWrapper, ScormManager, and DevController.

scorm.js

The web application (LMS) is responsible for launching and tracking SCOs that are created in Unity. These SCOs communicate with the ScormAPIWrapper in Unity to pass data back and forth between the LMS and the DLX project. the scorm.js library is a JavaScript API that provides a standardized way of communicating with an LMS from a web application.

ScormAPIWrapper

The ScormAPIWrapper is a Unity script that provides an interface between the web application and the DLX project. It is responsible for receiving data from the LMS and passing it to the ScormManager script, which handles the tracking and reporting of data to the LMS.

ScormManager

The ScormManager script is responsible for implementing the SCORM standard by acting as a connection between the code and the ScormAPI. The ScormManager sends the data to the ScormAPIWrapper, which includes launching SCOs, tracking progress, and reporting data back to the LMS. Overall, ScormManager handles the overall management and coordination of the communication and ensures that SCORM API calls are sent correctly and data is handled properly.

DevController

The DevController script implements and exposes a specific part of the SCORM API by calling a method from ScormManger. The DevController handles the actual communication and translation of the specific API calls (Ex. SetActivityComplete()) for the SCORM API. Additionally, the DevContoller allows developers to test and debug the SCORM API structure. It provides a way to simulate an LMS and test the communication between the LMS and the DLX project.

Compatibility with Unity

The SCORM standard is a guideline that defines a set of technical standards and does not have a preference for any specific technology, as long as the technology is SCORM-compliant. That said, tracking an embedded Unit project on an LMS can be compatible, but it depends on several factors:

  1. SCORM Compliance

Non-compliant content may not be trackable or may cause compatibility issues with the LMS. There are four different implementable versions of SCORM. SCORM 1.2 and SCORM 2004 are used most widely, and xAPI makes data collection possible.

SCORM Non-complaint Example

Using the SCORM API Structure does not mean being SCORM compliant. Here is an example:

A Unity project using the SCORM API Structure tracks completion and score but does not properly report them back to the LMS.

  1. Version of Unity

Some versions of Unity have built-in support for SCORM, while other versions may require additional plugins, extensions, or scripts.

SCORM support was added to Unity in version 5.6. However, the SCORM API Structure will still need to be implemented in later versions to ensure compatibility.

  1. LMS Compatibility

The LMS hosting the web application and the embedded Unity project should support the SCORM standard, but it is possible that LMSs that support SCORM have limitations or issues with tracking embedded content.

  1. Communication Protocol

The communication protocol used between the embedded Unity project and the LMS can also impact compatibility. If a different protocol or implementation is used, this may cause issues.

Different SCROM protocol Example

If an embedded project is designed to use SCORM 1.2, but the LMS supports SCROM 2004, this could cause compatibility issues with the tracking and reporting of data. In addition, embedded content using a custom implementation of the SCORM API structure may also cause issues.

Summary

Ensuring the compatibility of a Unity project embedded in a SCORM-compliant LMS is compatible requires following the specific guidelines and rules of the SCORM standard. By ensuring that the Unity project is SCORM compliant, the LMS is capable of supporting the SCORM standard, and the communication protocol is implemented correctly, compatibility between the DLX project and the LMS can be ensured. Compatibility allows proper tracking and reporting of data to be achieved.

Brightspace API

Overview

The Brightspace API is a set of REST (Representational State Transfer) web services that enable developers to integrate D2L Brightspace with other systems or applications. It allows for the exchange of data between D2L Brightspace and external systems, which can provide greater flexibility and customization options for users.

ACTION REQUEST

The Brightspace API allows clients to create, modify, retrieve, or delete resources used by an LMS. These actions consist of an HTTP URL and parameters and can be accessed directly or through a library that abstracts the URL manipulation. API calls must use HTTPS and comply with the same security requirements as accessing Brightspace through a browser.

STRUCTURE

Each action has an HTTP method (eg. GET, POST, DELETE), and a URL structure that includes the target host, LMS product component, API version, and path to the particular action.

target host

The host name for the LMS deployment (Ex. eConestoga).

component

A short string identifying an LMS product component.

version

API version supported by the product component.

path

Path to the particular action. This will have static structural information about the kind of action you’re taking, but may also include parameterized data like entity identifiers, or action types.

Compatibility with Unity

The compatibility of an embedded Unity project on eConestoga using the Brightspace API will depend on specific requirements and implementation. However, the authentication and security of user information.

Authentication and Security

OAuth 2.0 is the authentication protocol this is supported by the Brightspace API for user authentication. When using OAuth 2.0, a user needs to grant permission for an application (such as the DLX project) to access their data in Brightspace. Once the user grants permission, the application receives an access token that can be used to make requests to the API on behalf of the user.

The compatibility concern involves the implementation of OAuth 2.0 authentication in Unity, as the process is complex. There are several steps involved, including:

  • registering your application with the Brightspace API

  • Configuration of authentication flow in Unity

  • Handling the access token from user permission

More information about setting up and registering OAuth 2.0 can be found here

Summary

Brightspace API is a set of REST web services that allow developers to integrate D2L Brightspace with external systems. The API uses HTTPS and complies with the same security requirements as accessing Brightspace through a browser. The compatibility of an embedded Unity project with the Brightspace API depends on specific requirements and implementation, but authentication and security are important considerations.
