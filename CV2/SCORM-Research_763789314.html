Introduction

This page is being used for the LMS integration team to determine what SCORM can continue to offer us or if we should switch to the Brightspace API.

VARLab has already made SCORM documentation available here. This means we will not need to start from scratch creating documentation and will simply need to go through and edit it as we see fit.

SCORM Setup


After following the process of adding SCORM to a project, the process is very simple and hard to get confused by. It involves creating a ScormManager GameObject in the hierarchy and adding the SCORM scripts to that. There is already a tutorial from VARLab for this, available here.


The process for creating a SCORM object from a project also was very straightforward. There is, again already, documentation available from VARLab that can be found here. We will again just need to update it if necessary.

NOTE: I currently do not have access to the documentation for uploading the SCORM object to eConestoga, so I currently cannot speak on this process, we will need to try and get access to this process so I can go through it and talk about my experience through it

What does SCORM offer

Looking through the code for the CORE Sandbox, the following is what was found.

Every student has a Student Record, this class appears to have many valuable properties and contains classes for different student data. Below are some of the ones that seem useful in some capacity

Helpful Classes

StudentRecord class

This class contains various helpful enums that could be used per user, things such as CompletionStatusType, ModeType and EntryType, which all control a user's interaction with a learning simulation.

The StudentRecord class contains various other classes that are kept in that students record, the following are some of the ones that seem important:

LearnerPreference class

This class would be useful in allowing users to have their own preferences for the audio of the learning simulation. Could maybe be extended to include more things?

LearnerScore class

This class would be useful in allowing us to keep track of a user score for different objectives/interactions, as well as having custom min/max scores for different things.

CommentsFromLearner/CommentsFromLMS

Both of these classes handle messages from both the user and the LMS, which could prove useful depending on the type of learning simulation.


Objectives class
This class seems extremely helpful as we can create objectives for the user.

Other useful properties


All of the following are available properties of the classes stated above that could prove extremely useful for creating a more interactive and user-friendly application.

Useful Methods

The ScormManager class contains various methods we could use throughout our learning simulations. All of the useful methods are getters/setters for the classes stated above that are contained within StudentRecord. We can basically edit and get any of the classes as we see fit, so all of the methods could potentially be useful.

The Downsides to using SCORM

  • SCORM is really old compared to the Brightspace API.

  • Cannot do as much as we can with Brightspace API without adding functionality

  • Other VARLab developers have expressed their distaste with debugging SCORM, and that it can be tough to do

  • SCORM doesn’t cover everything we want/need out of an LMS, an example is not being able to email students/profs


The Upsides to using SCORM

  • Already been implemented within VARLab and partially understood by some Developers.

  • Ease of use.

  • SCORM does not only work with bright space, SCORM is universal, whereas the Brighspace API only works for Brightspace

  • We already have tutorials for most of SCORM, they may just need to be updated but not started from scratch.


Options

So what are our options if we choose to go with SCORM?

First Option - Use SCORM as it is
This is not the ideal solution, while SCORM alone offers some useful tools we can try and take advantage of, it does not have everything we need. The tracking seems basic and barebones and is also not currently set up. It does not have any support for sending out emails (which was something that was mentioned during our meeting with David). Basically, if we choose to go with this option, we are severely limiting what VARLab can track through learning simulations. This approach would probably take the least amount of time to implement but would result in us being unable to track very well.

Second Option - Extend SCORM with Experience API
This API was brought up during our research. It appears that we could be able to not only use SCORM but extend its functionality with this API, allowing us to do way more things while still using SCORM. If we decide to humour this approach, more research will need to be done on this API to see what it offers in full. This approach would take some time to both learn this API and implement it into our projects, but the possible outcome would be that we have new toys to play with when tracking stats through a learning simulation.

Third Option - Don’t use SCORM

The third option is that we do not use SCORM, and we instead use the Brightspace API that can replace SCORM. I do not know as much about it as I am not the one researching it, so in order to get an idea of if this is what we should use, please take a look at the Brightspace API research documentation.
