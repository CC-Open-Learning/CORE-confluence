Potential Options

Throughout this project, we have come across many options for LMS Integration. These options included:

  • SCORM

  • Brightspace API

  • Common Cartridge

  • LTI

each of these showed various levels of success in how they would work for VARLab, and the following is how we felt about each at the end

SCORM

Our research found that SCORM cannot give us everything we want. We already use SCORM objects in current DLX projects, but not much is done other than getting the students' IDs.

SCORM offers various features such as comments, progression, completion status and even time spent features.

Much was done to try and get more of the features of SCORM to work properly, but nothing seemed to work properly. Various things went wrong, data wasn’t sent, and in the end, it seems like it is not worth it to try and make it work.

Furthermore, after a discussion in the previously mentioned D2L meeting, we were again told that we will not be able to achieve what we want with SCORM, which further helped in making the final decision to move away from SCORM.

Unless something further happens with SCORM, it has been decided that we will not try to implement it further. If needed, the SCORM documentation can be found HERE, and our SCORM research can be found HERE.

Brightspace API

Brightspace API is the first-party API used in eConestoga. While eConestoga IT uses Javascript to make calls to the API, we would integrate it into Unity to make calls from the DLX projects.

Methods to integrate the API calls with Unity currently include .NET controller and Unity Web Requests.

.NET controllers require importing a .NET controller that will make the requests when we call its methods, making it easily expandable by including new methods for each endpoint. This method requires the .NET libraries that are currently problematic with using in Unity(see ticket: CORE-456).

Unity Web Requests is a Unity class made to create API Requests. Since it’s already part of Unity, it has no issues with integration in Unity. Unity Web Requests are not easily expandable, requiring each request to be coded from the ground up. Unity Web Request would be required if .NET cannot be supported by Unity.

Before Integrating with Unity, testing the API with JavaScript embedded in eConestoga will check if the request successfully calls the endpoint. Currently, JavaScript requests haven't been started.

Our Brightspace API research can be found HERE.

Common Cartridge

Common Cartridge is probably our least researched option, as we did not know about it for a while. It was first mentioned during the D2L meeting as a possible option. Therefore, little was known about it to determine whether it was a good choice.


It was known that it was developed by the same organization governing LTI. This means there is a lot higher chance for integration between LTI and Common Cartridge than with SCORM. This is only going to work if we decide to go with hosting our DLX projects as LTIs, and not SCORM objects.

Our Common Cartridge research can be found HERE.

LTI

LTI, at first, wasn’t one of our options. It wasn’t until we had the meeting with D2L that we considered it an option. It offers a lot of good things that we can potentially use. The problem is that no existing project hosts it as an LTI, furthermore, the only team that we know of that uses LTI is the VConestoga team, so if we were to try and host our DLX projects with LTI, it would basically be from scratch, but it may be a necessary risk in order to get more out of our LMS. This is the option we wanted to try and make work the most, if we had more time.

Currently looking into using our Moodle Instance to make a Moodle LTI to integrate into eConestoga. We don’t have credentials for the Moodle as of yet. David is looking into getting the credentials. Any questions about Moodle can be asked to David or Jacky as Jacky set up the Moodle instance. Links to Moodle LTI integration is at the bottom of page.
