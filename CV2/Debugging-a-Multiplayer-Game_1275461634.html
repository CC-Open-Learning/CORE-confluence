Introduction

Debugging a multiplayer game can prove to be very difficult and can become much more complicated than offline games due to the networking aspect, and this page aims to go over some of the ways that debugging a multiplayer game can be done, as well as some extra things that may need to be debugged due to the nature of multiplayer.

Unity has official documentation that goes over a lot of different ways debugging can be done; this document will not cover every method that Unity covers, so if none of these options provide a solution, please check out the Unity documentation.

If better/more ways of debugging are found, they can be added to this document.

Using a clone

Using a clone is probably the most useful thing when creating multiplayer games, allowing you to test the game with an actual 2nd person without the need to make a 2nd build. The two biggest packages that allow for this are ParrelSync and Multiplayer Play Mode.

By using these, you can test your game with 2 people, allowing developers to easily test and find network-related bugs.

These packages also make it very easy to make a change and immediately test again, if you find something that is wrong, something you should change, etc, you can just update the script on your main editor, and the clones will be automatically updated to reflect what the new change is.

The same can be said for anything changed in the scene hierarchy. When you make a change in the hierarchy and save it, you can reload the other editors to immediately reflect this change onto them.

Breakpoints

Breakpoints can be used in multiplayer games almost the same way they are usually used. One main difference is that they will cause players of the game to be disconnected because they fully stop the application from running while it is hit.

Because of this, if you use a breakpoint on a client, that client will be disconnected, and if you do it on the host, everyone will be disconnected after a certain amount of time. The amount of time this will take is set by the “Disconnect Timeout” setting on Unity Transport.

To avoid this, the Disconnect Timeout setting can be changed to allow for more time to elapse before people begin becoming disconnected. Just make sure it is set back to what it should be after testing.

One very important thing to know is if you’re using a clone from ParrelSync, you need to put the breakpoint on the clone that you want to stop on, if you want to break on Player 2 but put the breakpoint on Player 1s version of the script, nothing will happen when Player 2 triggers that code.

Logging

Logging can prove to be very beneficial in a multiplayer game; you can and should log basically everything, whether it be logging a crash or putting some debug logs to see what is happening; they will all prove beneficial when trying to overcome a bug or something of the sort.

One situation where something as simple as debug logs will prove very helpful is when trying to find out if an RPC is being called or if it's being called on the right person. Let us say we have a method being run on the server that calls another method on a specific client, in order to test that this actually works, you can add a debug log to the method and then use ParrelSync to make clones of the project, trigger this situation, and see if the debug log is ran on the correct client.

Simulated Network Connections

Another thing that can be done for both debugging and testing is utilizing the Multiplayer Tools package. This package offers a lot, including the following:

  • Network Scene Visualization - Overlays information in the scene view

  • Runtime Network Stats Monitor - Monitor bandwidth usage during runtime

  • Network Simulator - Simulate network conditions

  • Network Profiler - Monitor bandwidth usage of the game

  • Hierarchy Network Debug View - Adds Object Ownership information in the scene

Utilizing these together can be very helpful in both debugging and testing a multiplayer game
