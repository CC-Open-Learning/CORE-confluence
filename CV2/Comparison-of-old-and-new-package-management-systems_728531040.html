Weaknesses

  • All functions to interact with the system are available within the package manager window (as far as I know)

  • Allows easy reinstallation of outdated versions of packages through the git branches

  • Samples work as intended

  • UI integrated elements give a better visual representation of the properties and characteristics of a package upon first glance

  • Requires a setup for users to get SSH authentication before any packages can be accessed

  • Without proper permissions for individual repositories or packages, Unity will show an error before launching, even if the user is SSH authenticated (I ran into this with node canvas package since it isn’t part of com.varlab)

  • Multiple copies of the same version existing on different branches could cause some confusion and/or inconsistencies

  • Some command line involvement for setting up the SSH keys

  • There isn’t very much documentation on the process of interacting with this system

  • Setup only done at the beginning of a project, rather than needing to be consistently modified as new packages are added

  • Config files only need to be created once and they will work for all projects on a given computer

  • Older projects should be able to transfer to this system without too many issues regarding the package versions that they are using, even if the version they are on doesn’t exist in the registry

  • Packages are isolated in a specific location, which ideally will make managing them easier over time

  • Separates our custom packages into a separate window within unity to make things a little bit easier to navigate as more packages are created

  • Documentation for using this version is more extensive and all in one place

  • As far as I know, outdated package versions will be available in a dropdown under each package (assuming they have been published to the registry)

  • As of now, samples require a low-moderate amount of extra work to set up, and may require an alternative solution, which as a result would further complicate the workflow and may not pay off

  • Currently, there is still a lot of command line involvement for the people that manage the registry itself (adding, updating, deleting)

Both of the above concerns should be resolved by CORE-360

Key Takeaways

  • If the process of setting up SSH authentication was fully eliminated, the config process of setting up a computer for accessing the VARLAB custom package registry would be shorter and guarantee that projects only need to set it up once per project, rather than consistently having to manually add new packages as new mechanics are implemented

  • Ideally, the new system would eventually host all of CORE’s custom and third party packages in the Verdaccio registry

  • In order to increase scalability of our systems overall, it is near essential that we begin storing our packages externally so that project production doesn’t begin to slow down
