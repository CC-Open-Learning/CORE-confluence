Commit ID

A ‚ÄúCommit ID‚Äù is an ID given to every commit. It provides a way to identify every commit separately. When clicked, it directs you to a commits page on Bitbucket to see all of the commit‚Äôs information.


When viewing a list of commits, click on the link that appears to be a random collection of numbers and letters. This is your commit's randomly generated ID. It should look similar to the one in Figure 1.

A page with all the information on that specific commit is shown when this is clicked.

Date of Test

The date of test is the date that a developer ran the specified test. It is essential to keep track of this for all of your tests for traceability.

Code Coverage %

Code Coverage % is the amount of code covered in your tests, displayed as a percentage. This percentage is found in the generated HTML report document. You can find the code you created and see how much of it was covered by the tests you wrote.

Figure 2 shows the report's overall summary statistics. The Line Coverage measures how many lines have been covered by the tests that ran, throughout the entire project. Line coverage will be what you will report in your project‚Äôs code coverage document. While this can be a useful measurement, it‚Äôs important to emply a risk-based code coverage strategy.

You can see the stats for each specific class. Figure 3 shows the stats for the mainMenu class. It has achieved 35.7% code coverage. You can click on an individual class to see which lines were hit or missed.

Most code coverage tools use highlighting to display results alongside the percentage. The code highlighted in green means that your tests successfully hit it. If it is highlighted in red, it means that your tests did not hit that section of code.

In the mainMenu class (Figure 4), we can see that different lines have been highlighted in red and green. This is because only parts of this class were hit by our tests. We would need to make more tests to achieve a higher code coverage in this class.

Untestable Lines

Untestable Lines are code lines that an automatic test can not cover. This can vary and depends on the project being worked on, but it is crucial to document any code found to be untestable, and why it is untestable. Future developers will know that it is untestable and plan accordingly if necessary.

Refer to the Testable Behaviour document for more on what is considered untestable.


Automatic Testing

Automatic Testing refers to tests that are run automatically. You write a test to do something in the project, run the test, and Unity will automatically perform the test you wrote.

Figure 5 shows an example of a simple automatic test. This test instantiates an asteroid, moves it, and then checks if the asteroid was moved. The test will return as either Passed or Failed. This is all done automatically.

Manual Testing

A manual test is a test that needs to be done manually by a tester. Although automatic tests are usually better, sometimes, especially when using Unity and developing games, there may be something that is too complicated or time consuming to automate. This is when a manual test is ideal.

With Unity, you can record your project using the Code Coverage package, start your game, and perform whatever you want to test. You can then end the recording and see the same results as if you had done it automatically. It would be best if you strived to make tests automatic, but be prepared to create some manual ones when needed.

Tested File - From Code Coverage Template

This is the name of the file that you are writing tests for. It is essential to keep track of this for traceability.

Related Tests - From Code Coverage Template

The related tests field shows which tests cover which lines in a class. In Figure 6, TEST-SAND-008 covers lines 0-125, and TEST-SAND-009 covers lines 151-200. It is essential to keep track of this for traceability

Reasonings - From Code Coverage Template

The reasonings column is used when we cannot test certain lines of code. We need to document the reason why we cannot test a line of code. This helps us try to fix it, get around it, or learn and teach future developers that this code is untestable. Figure 6 shows an example of untestable code. We specify the file the code is in, which lines, and why we cannot test it.

Cyclomatic Complexity

Cyclomatic Complexity helps us determine how complex our code is. This tool is valuable for refactoring our code and making it less complicated. It also tells us how many tests we should write to test every path of our function correctly.

You should rarely need to figure out your function's Cyclomatic Complexity manually. However, knowing how it is calculated for learning is essential.

There are two ways to find the Cyclomatic Complexity of a function. The first way is to use the formula Cyclomatic Complexity = Edges ‚Äì Nodes + 2(Exit Points). More about this formula can be found¬†here.

The second way is to look at the function and count the number of decisions in the function. If, else if, and else statements, as well as loops, all count as 1 for the complexity. Additionally, the function declaration is also considered as 1.

The automatically calculated complexity of your function can be found in your code coverage document.

Figure 9 shows the standard for Cyclomatic Complexity. This is a good rule of thumb for your project. However, this can be altered per your team‚Äôs needs.

Note: Cyclomatic Complexity is much more complex then what is written in this documentation. This documentation is kept simple to be easily understood. If interested, look more into it, as it is an interesting topic.

CRAP Score

A CRAP score is a metric used to describe how good or bad a section of code (specifically a method) is. This term was created to rate code with very broad reasoning, like poor testing or high complexity. The formula used to calculate CRAP score is as follows:

CRAP1(m) = comp(m)^2 * (1 ‚Äì cov(m)/100)^3 + comp(m)

Where CRAP1(m) is the CRAP1 score for a method m,

comp(m) is the cyclomatic complexity of m,

and cov(m) is the basic path code coverage from automated tests for m.

If CRAP1(m) > 30, the method is generally considered "CRAPpy." Aiming for 30 is a good rule of thumb. However, referring to your project about what should be considered acceptable levels of "CRAPpy" code is preferred.

NPath Complexity

The NPath Complexity of a function refers to how many different outcomes the function has. The following code snippet provides an example.

 public static string getDialogue(int choice)
        {
            if (choice == 1)
            {
                return "You purchased a potion!";
            }
            else if (choice == 2)
            {
                return "You purchased a mana potion!";
            }
            else 
                return "You did not purchase anything";          
        }

This function receives an int and returns a dialogue based on the number. Since this function has three possible return statements it can hit, it has an NPath Complexity of 3.

Note: As of writing this, NPath Complexity in unity does not currently work and will always return 0.

Unit Test

A Unit Test is a test that checks an individual unit of code. You typically want each test only to test one specific thing, meaning unit tests should only have one Assert statement. Unit tests usually use the Arrange, Act, Assert formula.

  • Arrange - where you arrange everything your test will use, such as variables or objects.

  • Act - where you perform the test, such as adding two numbers.

  • Assert - where you check if the test passed or failed, such as checking if adding two numbers gave the correct result.

The following code snippet is a simple example of a Unit Test that checks if two numbers are added correctly using the Arrange, Act, Assert formula.

Note: This is a basic C# Unit Test, not a Unity test.

[TestMethod]
public void addTwoNumbers_GivenTwoNumbers_CorrectNumberReturned()
{
  //Arrange
  int numberOne = 1, numberTwo = 2, expectedResult = 3;

  //Act
  int actualResult = MyClass.addTwoNumbers(numberOne, numberTwo);

  //Assert
  Assert.AreEqual(expectedResult, actualResult);
}
