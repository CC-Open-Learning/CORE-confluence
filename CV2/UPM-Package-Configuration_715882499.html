Summary

Unity uses a package manifest file (package.json) to manage information about a specific version of a specific package. The package manifest is always at the root of the package and has crucial information about the package, such as its registered name and version number.

The package manifest also defines useful information to communicate to the user, such as:

  • A user-friendly name that appears in the UI.

  • A brief description of the package.

  • The earliest version of the Unity Editor the package is compatible with.

https://docs.unity3d.com/Manual/upm-manifestPkg.html

Package Configuration

The package manifest is located in Assets/Package/package.json within a project’s file structure.

Much of this page’s content has been taken directly from the Unity documentation on the UPM package manifest structure, and then modified to describe the specific requirements for CORE Framework.

Required Properties

If these properties aren’t present, either the registry refuses the package when it’s published, or the Package Manager can’t fetch or load the package.

Property

JSON Type

Description

name

string

A unique identifier that conforms to the Unity Package Manager naming convention, which uses reverse domain name notation. For more information about the naming convention, refer to Naming your package.

Note: The name identifier is different than the user-friendly name that appears in the list view in the Package Manager window.

version

string

The package version number (MAJOR.MINOR.PATCH).

For example, “3.2.1” indicates that this is the 3rd major release, the 2nd minor release, and the first patch.

This value must respect Semantic Versioning. For more information, refer to Versioning.

publishConfig

array of strings

While not technically a required field for a UPM package, CORE Framework packages must specify the publishConfig array in order for packages to be correctly published to VARLab’s scoped registry for UPM packages.

See the example package manifest below for the appropriate usage of the publishConfig field.

Recommended Properties

The Package Manager can install packages in a project even if the recommended properties don’t have valid values or are missing.

The recommended best practice is to assign values for these properties to ensure that your package is discoverable and to offer a better experience for users.

Property

JSON Type

Description

description

string

A brief description of the package. This is the text that appears in the details view of the Package Manager window. This field supports UTF–8 character codes. This means that you can use special formatting character codes, such as line breaks (\n) and bullets (\u25AA).

displayName

string

A user-friendly name to appear in the Unity Editor (for example, in the Project Browser, the Package Manager window, etc.).

For example, Unity Timeline, ProBuilder, In App Purchasing.

unity

string

Indicates the lowest Unity version the package is compatible with. If omitted, the Package Manager considers the package compatible with all Unity versions.

The expected format is “<MAJOR>.<MINOR>” (for example, 2018.3). To point to a specific patch, use the unityRelease property as well.

Note: A package that isn’t compatible with Unity doesn’t appear in the Package Manager window.

Example Package Manifest

The package.json manifest below, taken from the https://bitbucket.org/VARLab/package-template/src repository, illustrates a complete list of the required and recommended fields for a CORE Framework package configuration.

{
	"name": "com.varlab.packagetemplate",
	"displayName": "UPM Package Template",
	"version": "1.2.3",
	"unity": "2022.3",
	"description": "An example of a UPM package which can be hosted in a scoped registry.",
	"category": "Instrument",

	"author": {
		"name": "Packaging Pete",
		"email": "pete@petesprofessionalpackagingparlor.io",
		"url": "https://varlab-dev.atlassian.net/wiki/spaces/CV2/overview"
	},

	"keywords": [
		"unity", "editor", "upm"
	],
	
	"samples": [
		{
			"displayName": "Example Project",
			"description": "Contains a sample for the package",
			"path": "Samples~/Example Project"
		}
	],

	"publishConfig": {
		"registry":"http://upm.core.varlab.org:4873/"
	}
}

Publishing and Updating Packages

Updated package versions are published automatically using Bitbucket’s continuous integration pipelines, as long as the project repository has been configured with the UPM_TOKEN repository variable, which requires an authentication token from the packagingpete development account.

More details on these integration steps can be found in the https://bitbucket.org/VARLab/package-template/src/main/CI/ folder of the https://bitbucket.org/VARLab/package-template/src/main/ repository.

Contact an IT Ops team member for support with publishing and unpublishing packages.
