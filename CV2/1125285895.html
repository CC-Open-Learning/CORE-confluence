This page is not meant to be a comprehensive beginner tutorial, however it will cover some basics in the beginning and gradually get less detailed towards the end. This page is intended to teach you how to approach implementing new UI in UI toolkit once you are familiar with the bare minimum. I recommend viewing the page Introduction to UI Toolkit and the resources at the bottom of that page for beginner tutorials.

How to Approach Creating New UI

  1. Create a blank UXML document by Assets/Right Click > Create > UI Toolkit > UI Document

  2. (Optional) Box your UI

    1. The first step to implementing UI is structuring your hierarchy. Styling and functionality can come afterwards. Your hierarchy will usually be full of invisible wrapper classes and parent VisualElement objects that simply exist to shift elements around in your UI. A good way to help visualize these elements is by boxing your UI design. This step is optional. It can help developers who not familiar with UI Toolkit, but is by no means required. What I mean by this, is to literally draw boxes around the elements to see the building blocks of VisualElements/<divs> in your UI.

Default:

Boxed:

  1. Red: The parent most VisualElement that wraps the entire modal. This has the background colour and alignment rules for children elements

  2. Orange and Yellow: Separate the parent into the larger usable blocks such as Header, Content, and Footer that have different justification/alignment

  3. Blue: Name label of the modal in the Header block

  4. Indigo and Purple: Title and SubTitle label of the modal

  5. Pink: Primary and Secondary button actionable components of the modal

  6. Brown: Note label component of the modal

  1. Structure your UXML

  2. Style your UXML through USS classes. These should point to colour variables in the Dark/Light theme files

  3. Write your script

UI Builder Setup

  1. Set your canvas larger than default. I like to set it to 1920x1080. You can also set it to match your editor window.

  1. Link your required USS filles. Most UI will have velcro-structure.uss (margin, positioning, layout classes), velcro-style.uss (colour, border, typography classes), and their specific sheet for this it will be amb-modal.uss.

Structuring & Styling UI

I’ll be using USS classes from velcro-structure.uss, and velcro-style.uss to do this. These classes can be applied to elements by dragging them from the USS class selector panel on top of the element in the hierarchy panel or by typing in the class name in the inspector and pressing “Add Style Class to List”.

1. Using Templates as a Starting Point (Red/Orange/Blue Boxes)

VELCRO has many UXML Templates that can be used as building blocks or starting points for UI. The modal header and background are one of those. To start, go to the Project tab in the Library panel, and drag in the “Window-SingleBackground” template from “/Assets/VELCRO UI/Templates/Header and Windows/”.

When in the hierarchy, Right Click > Unpack Instance. You cannot add child elements or make changes to a template container. It needs to be unpacked. Move the “#Canvas” element out of the “#Window-SingleBackground” element and delete it.

Templates are wrapped in an empty VisualElement when unpacking the instance. For this example, it is element “#Window-SingleBackground”. These can be deleted

The templates are already set up with support for light and dark mode themes. No USS classes are needed for these elements unless you want to change the template's default. It is also important to keep templates packed such as the header when possible. Packed templates will maintain the reference and receive update from future VELCRO versions. The modal template needed to be unpacked as we will be adding additional content to the window.

2. Setting Up the #Content Container (Yellow Box)

For the content container, we set it to take up all available space with flex-grow: 1, and to align all child elements in the center. When boxing the original design, all labels and buttons were horizontally centered. This is a good indication that they can all share the same parent wrapper to have alignment set only once. 

.grow {
    flex-grow: 1;
}

.align-items-center {
    align-items: center;
}

3. Adding Labels to #Content Container (Purple/Indigo Boxes)

All the alignment and justification for the title has been done in the content container. All that’s left is styling the text by removing the margin/padding, making it bold, increasing font size, and setting colour.

--colour-text-primary: #FFFFFF;

m-0 {
    margin: 0px;
}

p-0 {
    padding: 0px;
}

.color-text {
    color: var(--colour-text-primary);
}

.fs-36 {
    font-size: 36px;
}

.bold {
    -unity-font-style: bold;
}

Similar to the title label, all the alignment and justification has been done for the sub title from the content container. All that’s left is to style with font size, italic, and colour.

I removed all margin from the title and sub title before applying a 24px margin to 1 element to separate them. When it comes to styling, there are many different routes to the same outcome. I prefer to lump margins onto as few objects in the hierarchy as possible to make debugging easier, but that is not the only solution.

--colour-text-secondary: #B3B3B3;

.color-text-secondary {
    color: var(--colour-text-secondary);
}

.fs-20 {
    font-size: 20px;
}

.italic {
    -unity-font-style: italic;
}

.m-0 {
    margin: 0px;
}

.p-0 {
    padding: 0px;
}

.mt-24 {
  margin-top: 24px;
}

4. Adding Buttons (Pink Box)

The buttons are side by side, and sibling VisualElements stack vertically on top of each other by default. Drag another container VisualElement into the hierarchy as a child of Content.

The direction needs to be set to flex-row, the buttons need to be horizontally centered, and the container should span the entire width of the parent. Next, drag in a button-secondary-large-1, and button-primary-large template from the Project tab in the Library panel. The buttons need an 8px margin and the text changed but the rest is already created from VELCRO.

.flex-row {
    flex-direction: row;
}

.align-items-center {
    align-items: center;
}

.justify-center {
    justify-content: center;
}

.grow {
    flex-grow: 1;
}

.w-100 {
    width: 100%;
}

Buttons and other controls are where BEM (Block-Element-Modifier) naming convention can be introduced. This should be used where possible. It’s useful to create parent-child relationships between classes through names rather than complex selectors that reduce performance.

  • Block (The parent container that is a piece of UI or control)

  • Element (The element you are modifying that is part of the parent block)

  • Modifier (The specific variant of the element. Think of this as a prefab variant)

In this example below for the modal, the block is “button”, with the element “primary” or “secondary”. A modifier for these classes would be something such as “.button-primary-disabled”. For these buttons, the base “.button” class has all USS that both buttons require. This makes “button-[element]” a simple change to the background colour in “primary” and “secondary”.

--colour-button-primary: #FFC009;
--colour-button-secondary: #E1E1E1;

.button {
    margin: 0;
    border-radius: 10px;
    padding-left: 40px;
    padding-right: 40px;
    padding-top: 12px;
    padding-bottom: 12px;
    -unity-font-definition: var(--lato-bold);
}

.button-large {
    width: 300px;
    height: 62px;
    font-size: 28px;
}

.button-primary {
    border-width: 0;
    background-color: var(--colour-button-primary-default);
    color: var(--colour-text-primary-default);
}

.button-secondary-1 {
    border-width: 2px;
    background-color: var(--colour-button-secondary-1-default);
    border-color: var(--colour-button-secondary-1-border-default);
    color: var(--colour-text-secondary-1-default);
}

5. Adding the Note label (Brown Box)

Drag in another label element from the library into the hierarchy and simply change it’s font size and colour.

--colour-text-tertiary: #21D8FF;

.color-text-tertiary {
    color: var(--colour-text-tertiary);
}

.fs-18 {
    font-size: 18px;
}

.m-0 {
    margin: 0px;
}

.p-0 {
    padding: 0px;
}

.mb-36 {
    margin-bottom: 36px;
}

Summary

To summarize the UI implementation, there are many ways to achieve the same outcome for a design of UI and this is not the best solution. This was intended to get you on the right path on how to start thinking about implementing new UI and give a good example, not the only example.

The important takeaways from this page are:

  • UI should be broken up into large containers that exist simply to position your content in the UI

    • A good strategy is to have Canvas/Header/Content/Footer containers in most UI

    • This can be visualized easier by boxing your UI

  • Elements that have more complex classes should use BEM (Block-Element-Modifier) naming convention to avoid complex selectors if possible

  • USS classes should never have hard coded HEX/RGB/Font values as it stops light/dark mode support and is bad practice. USS variables should be used

  • USS building block classes from velcro-structure, and velcro-style should always be used when possible. For this example, new classes should only be created in a new stylesheet specifically for this modal, modal.uss for properties that don’t already exist
