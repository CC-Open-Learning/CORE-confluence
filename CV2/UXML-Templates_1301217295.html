Controls

Templates are UXML files that have no script functionality behind them. They are completely assembled UXML files that have all required USS classes/VisualElements applied. They can be dragged & dropped into the UI Builder for quicker implementation. Using Templates is highly recommended.

Templates can be found under Library Panel > Assets > VELCRO UI > Templates

Toggles

https://www.figma.com/design/om9GwMhkfeJpcADbKz4ov8/VARLabs-Design-System?node-id=1709-18908&t=K1P2UOha0HdkhO3p-1

Toggles are a custom control. They are a Boolean based UI similar to checkboxes . Unlike other templates, toggle has a script defining it’s behaviour SlideToggle.cs. Toggles cannot be created by taking a default Unity control like buttons and applying USS classes, for these, you must use the UXML template provided by VELCRO.

Toggle can be added by dragging & dropping Toggle.uxml in Assets > VELCRO UI > Templates > Toggle > Toggle.uxml

Scripting

Setup

  1. Create a reference to the toggle where needed

private SlideToggle toggle;

private void Start() {
    toggle = root.Q<TemplateContainer>("Toggle").Q<SlideToggle>();
}

When querying for templates (Other UXML files that are in the current UXML file), then you must first query for type Q.<TemplateContainer>(“Name”), then requery for the element type you are trying to access

Callbacks

  1. Register a callback where needed

// Subscribe the OnToggleValueChanged method to the toggle's value changed event.
toggle.RegisterValueChangedCallback(OnToggleValueChanged);

private void OnToggleValueChanged(ChangeEvent<bool> evt)
{
    // Logs the updated value of the toggle.
    Debug.Log(evt.newValue);
    
    // Logs the previous value of the toggle before it was updated.
    Debug.Log(evt.previousValue);
}

  1. To change the value WITHOUT invoking the <ValueChangedEvent>, call

slideToggle.SetValueWithoutNotify(bool newValue);

Properties

slideToggle.value = true; // Toggle is enabled.
slideToggle.value = false; // Toggle is disabled.

Sliders

https://www.figma.com/design/om9GwMhkfeJpcADbKz4ov8/VARLabs-Design-System?node-id=4956-2450&t=RKyXNk3IMrrCoitX-1

Sliders are a float based custom control. Unlike other templates, slider has a script defining additional behaviour FillSlider.cs. Sliders cannot be created by taking a default Unity control like buttons and applying USS classes, for these, you must use the UXML template provided by VELCRO.

Sliders can be added by dragging & dropping Slider.uxml in Assets > VELCRO UI > Templates > Slider > Slider.uxml

You can modify the slider’s properties by accessing the FillSlider element inside of the slider UXML. These will be visible in the inspector on the right side of the UI builder window.

Scripting

Setup

  1. Create a reference to the slider where needed

private FillSlider slider;

private void Start()
{
    slider = root.Q<TemplateContainer>("Slider").Q<FillSlider>();
}

When querying for templates (Other UXML files that are in the current UXML file), then you must first query for type Q.<TemplateContainer>(“Name”), then requery for the element type you are trying to access

Callbacks

  1. Register a callback where needed

// Subscribe the OnSliderValueChanged method to the slider's value changed event.
slider.RegisterValueChangedCallback(OnSliderValueChanged);

private void OnSliderValueChanged(ChangeEvent<float> evt)
{
    // Logs the updated value of the slider.
    Debug.Log(evt.newValue);
    
    // Logs the previous value of the slider before it was updated.
    Debug.Log(evt.previousValue);
}

Properties

slider.value = 5;       // The current value of the slider.
slider.lowValue = 0;    // The minimum value of the slider.
slider.highValue = 10;  // The maximum value of the slider.

Headers & Window Backgrounds

Header and window templates are barebones background UXML files to quicken the implementation of new UI in DLX. They have no functionality behind them and are simple UXML files that have a lot of the boilerplate structure complete.

How to Use

  1. Create a new UXML file by Right Click/Assets > Create > UI Toolkit > UI Document

  2. Drag and drop the desired template into the hierarchy

  1. On the template container, Right Click > Unpack Instance (Only unpack if you need it to make edits. Otherwise unpacking will break all references for VELCRO updates)

    1. Unpacking templates is just like unpacking a GameObject prefab in the hierarchy panel in the Unity scene

    2. You cannot add child elements to a template container, so if you are using these templates as a starting point for new UI, you must unpack to make further edits

  1. Move the first child element out, and delete the wrapper parent VisualElement. When unpacking, Unity adds a wrapper VisualElement that is not needed and may break some positioning

When you unpack a TemplateContainer, all USS files referenced in the template will now be referenced in your new UXML file under the StyleSheet panel

Main Menus

The Main Menu is currently a UXML Template only, with basic script functionality planned for a future version of VELCRO. It is a container designed to hold at least 2 instances of VELCRO Cards Versions 1 to 4, though primarily designed with Version 3 in mind.

Small Main Menu

https://www.figma.com/design/om9GwMhkfeJpcADbKz4ov8/CVRI-Design-System?node-id=6092-3048&m=dev

How to use

To get started with implementing the Main Menu template in your scene:

  1. Create a new UXML file by Right Click/Assets > Create > UI Toolkit > UI Document

  2. Drag and drop the MainMenuSmall template into the hierarchy

  1. On the template container, Right Click > Unpack Instance

    1. Currently, unpacking is required unless you are using the default 2 instances of Card Version 3 (as shown above). If adding more cards and/or changing the type of card used, you must perform this step. Otherwise, leave the template container intact.

    2. Unpacking templates will break references for VELCRO updates. If you unpack the Main Menu and use it, ensure you check the VELCRO template for changes when you update.

  2. Move the first element under the parent “#MainMenu” element out, and delete the now empty #MainMenu element wrapper, as it is no longer needed.

  3. Customize labels and cards as needed

    1. To customize the cards used in the menu (type and/or amount), simply add/remove card templates from the ‘Body’ element.

Node Steppers

Node steppers are a UXML Template only. This piece of UI will be a component of a larger UI such as RT’s handbook. They can be found at “/Assets/VELCRO UI/Templates/Steppers/” or in the Library panel in the UI builder by switching to “project”

How to Use

There are USS classes defined for each state, as well as UXML templates. Either dynamically clone the required templates at runtime, or drag & drop the required templates into your UI and switch classes at runtime.

  1. If switching classes:

    1. #Node needs ONE OF .node-completed, .node-in-progress, .node-not-started

    2. #Sprite needs ONE OF .node-sprite-completed, .node-sprite-in-progress, .node-sprite-not-started

    3. #Step needs ONE OF .node-step-completed, .node-step-in-progress, .node-step-not-started

  2. Steps will grow to take all available space. They have no defined width in px. As many steps as needed can be added.

    1. The TemplateContainer breaks the grow functionality. The step templates either need to be unpacked, or require the .grow class

Arrow Steppers

Arrow steppers are a UXML Template only. This piece of UI will be a component of a larger UI. They can be found at “/Assets/VELCRO UI/Templates/Steppers/” or in the Library panel in the UI builder by switching to “project”

How to Use

There are USS classes defined for each position, and state, as well as UXML templates already set up. Either dynamically clone the required templates at runtime, or drag & drop the required templates into your UI and switch classes at runtime.

  1. If switching classes:

    1. #Background needs ONE OF .arrow-background-completed, .arrow-background-in-progress to change styling

    2. #Background needs ONE OF .arrow-background-start, .arrow-background-middle or .arrow-background-end to change images

    3. #Border needs ONE OF .arrow-border-completed, .arrow-border-in-progress to change styling

    4. #Border needs ONE OF .arrow-border-start, .arrow-border-middle or .arrow-border-end to change images

    5. #HeaderLabel needs ONE OF .arrow-label-completed, .arrow-label-in-progress

    6. #DescriptionLabel needs ONE OF .arrow-label-completed, .arrow-label-in-progress

  2. Arrows will grow to take all available space. They have a minimum width of 400px

    1. The TemplateContainer breaks the grow functionality. The arrow templates either need to be unpacked, or require the .grow class
