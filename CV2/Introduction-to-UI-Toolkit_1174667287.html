Before Beginning with UI Toolkit

Before diving into VELCRO, it is important to understand that Unity’s UI Toolkit is heavily influenced from modern front-end web development and uses flexbox for layout. This is completely different from the UGUI system most Unity developers will be familiar with. Familiarity with simple HTML structure, CSS, and flexbox will be a huge asset in learning UI toolkit quickly.

 

Key Differences between UI Toolkit and Unity UI

Category

UI Toolkit

Unity UI

How UI is built

Uses a hierarchy of VisualElements similar to <div> and other HTML tags that make up a visual tree. This is edited in the UI Builder and results in a UXML file.

Uses a hierarchy of GameObjects where each game object is visible in the hierarchy editor panel.

Debugging

Debugging is done through a separate window (Window > UI Toolkit > Debugger) that is similar to you pressing F12 in your browser to view the “elements” tab.

Debugging is done through the game objects in the hierarchy.

Where UI lives

UI is referenced in a UIDocument component in the SourceAsset field on an empty GameObject. It also references a PanelSettings asset which handles various settings such as resizing.

All UI must be placed under a Canvas/CanvasScaler that controls scaling.

Element alignment/justification

UI Toolkit uses flexbox to control self alignment, child element alignment, and content justification through USS classes.

Unity UI uses a combination of Horizontal/Vertical/Grid LayoutGroup, LayoutElement, and ContentSizeFitter components to control self and child size/alignment.

UI Toolkit Components & Windows

  1. UXML file is what contains the UI Element hierarchy and all applied styles to each element either through USS classes or inline styles.

    1. Right Click > Create > UI Toolkit > UI Document

  2. USS file (Unity Style Sheet) is what contains all the classes that effect element look, behaviour (click and hover effects), and layout of each element.

    1. Right Click > Create > UI Toolkit > Style Sheet

  3. TSS file (Theme Style Sheet) is an asset that creates a relationship between other TSS and USS files. It can inherit styles from other TSS files (Required to inherit from UnityDefaultRuntimeTheme) and determines what USS files have visibility of what themes/classes.

  4. UI Document is the component that goes onto an empty game object in the scene. This associates a C# script with a UXML file, the PanelSettings asset and determines the sort order (z-index) of one UI compared to another.

    1. Add Component > UI Document

  5. PanelSettings is an asset object that defines shared behaviour for scaling and other settings of all UI that reference this asset in the UI Document component. A UI Document component requires a reference to a panel settings asset. You can have more than 1 panel settings asset.

    1. Right Click > Create > UI Toolkit > Panel Settings Asset

  6. TextSettings is an asset object that defines shared behaviour for font and other settings of all UI associated with this asset. The text settings asset is referenced in the panel settings asset.

    1. Right Click > Create > UI Toolkit > Text Settings

 

UI Builder

Window > UI Toolkit > UI Builder

All UI in UI Toolkit is created in the UI Builder. It is a visual editor where you can also preview and test any changes that don’t rely on influence from your associated C# scripts.

 

  1. The stylesheet (USS class list) selector. This is where you link desired style sheets to your current UXML document.

  2. Hierarchy. This is the tree view of your UXML document representing the UI Element hierarchy.

  3. Library. This is where you add preset UI templates like the default Unity button, slider, etc. to your current UXML document hierarchy. You can also add custom UI templates that are part of your current project.

  4. Viewport. This is where you view the UXML document in it’s current state. You can also preview pseudo state effects such as hover and click effects by clicking the “preview” button in the top right of the viewport.

  5. UXML and USS Preview. This shows you the underlying UXML code behind the UI elements that you drag into the builder as well as the USS classes associated with each element. However, all of this information is easier to understand in different locations. You can collapse this tab all the way down and ignore it most of the time.

  6. Inspector. This is where you change the properties of the currently selected UI element such as USS classes or inline styles.

 

UI Debugger

Window > UI Toolkit > Debugger

You cannot see the UXML document hierarchy at runtime through Unity’s GameObject hierarchy panel. You must use the UI debugger. This is similar to you pressing F12 in your browser to see the “Elements” tab. It will show the UI element hierarchy and the USS classes/inline style applied to each element.

 

  1. Hierarchy. This panel is where all UI Document hierarchies will be visible that are in the scene. Here you can see the ambulance project has separate UI Documents for a Menu, Settings Menu, Information Popup, etc.

  2. Layout/Box Model. This section is what will show the margin, border, padding, and element width in pixels for the currently selected element. This view is exactly like your browser view when pressing F12. If elements aren’t aligning correctly, this should be your first location to look.

  3. Styles. This section is what will show all the applied USS classes as well as the final resolvedStyle applied to the element after inline styles. You can also see the applied USS classes for each element listed in the hierarchy in yellow text. For the currently selected element in the photo, the “.background” and “.ambulance-theme” USS classes are visible in both the hierarchy tab and the “Classes” list in the styles section.

Note: You can change these values at runtime in the debugger to test changes before applying to the UXML document in the UI Builder.
