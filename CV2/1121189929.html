Before we get into creating our first piece of UI in VELCRO, it‚Äôs important to understand where the tools are and why the package is structured the way it is.

Project Structure

USS classes in VELCRO will only be in 2 forms. Small bite sized classes that mostly only ever change 1 property (Ex. Changing the alignment of an element from flex-start to flex-end), or complete classes that when applied, change an element from the default Unity theme to a ready to use VELCRO control (Ex. A primary button that changes size, padding, margin, border properties, colour, hover effects, etc.).

  1. Building Block Classes that only change ~1 property is what you will use most of the time when creating and structuring new UI in a DLX that is not included in VELCRO.

    .align-items-start {
    align-items: flex-start;
}

  2. Complete Classes are what you will use when you want to add the standard VELCRO look of a UI control (button, slider, etc.) to your new UI in a DLX. Alternatively, you can also drag in the template UXML document into your UXML document.

    /* Shared between all buttons in VELCRO */
.button {
    margin: 0;
    border-radius: 10px;
    padding-left: 40px;
    padding-right: 40px;
    padding-top: 12px;
    padding-bottom: 12px;
    -unity-font-definition: var(--font-bold);
}

/* Shared between all large buttons in VELCRO */
.button-large {
    width: 300px;
    height: 62px;
    font-size: 28px;
}

/* Applies the styles specific to primary buttons only */
.button-primary {
    border-width: 0;
    background-color: var(--colour-button-primary-default);
    color: var(--colour-text-primary-default);
}

All USS classes affecting the structure of UI elements (Ex. layout, justification, alignment, size, margin, etc.) will be contained in the USS file velcro-structure.uss. All USS classes affecting the look, feel, and behaviour of UI elements (Ex. colour, typography, border properties, hover effects, etc.) will be contained in the USS file velcro-style.uss. All USS classes that turn default unity theme controls (Ex. button, slider, etc.) into the standard VELCRO look will be contained in the USS file velcro-components.uss. Each separate template UI (Ex. Modal, Notification, Toast) will have it‚Äôs own USS file ‚Äúvelcro-[UI Name].uss‚Äù.

Performance

UI Toolkit is less performant than Unity UI on startup as of 2015. Unity has stated they want to improve performance starting in Unity 2022, but who knows if that has already happened to match Unity UI or if it‚Äôs coming in 2035 + 15 years. In the meantime, there are ways we can mitigate this.

For any sim that VARLab creates, performance from UI realistically won‚Äôt be an issue, but we should still be mindful of our decisions. The UI that DLX require is simple enough where we won‚Äôt run into most/all of these issues so we can ignore a little for the ease of development.

Each USS file linked to your UXML document is turned into a lookup table. The number of elements in the hierarchy is the main fact that affects performance. Performance decreases linearly as you add classes to an element. Each class in the list is used to query the lookup table. The complexity is N1 x N2, where:

  • N1 is the number of classes on the element

  • N2 is the current number of applicable USS files linked to your UXML document

So why VELCRO is designed around building block USS classes that only affect 1 property and split into multiple USS files? That jumps both N1 and N2 up a bunch, but this really only begins to be an issue for large UI that has several thousand elements. The answer is reusability and clarity. This is one of those rules that we can ignore for the ease of development.

Ex. Think of it this way when creating UI.

Scenario 1: You have to create a modal that pops up when you finish finding all objects in the scene that says ‚ÄúCongratulations! Would you like to replay the sim?‚Äù. Text is centered at size 28px and has an 8px margin.

.scenario1 {
  -unity-text-align: center;
  font-size: 28px;
  margin: 8px;
}

Scenario 2: You now have to create a new modal that pops up if the learner tries to leave the sim early that says, ‚ÄúAre you sure you want to quit?‚Äù on one line and a second line that says, ‚ÄúYour progress will not be saved.‚Äù. For this modal, the font is smaller due to more text, left justified, and has some additional styling.

.scenario2 {
  -unity-text-align: left;
  font-size: 24px;
  margin: 0px;
}

.warning-text {
  font-size: 18px;
  -unity-font-style: italic;
  color: red;
}

If you try to reduce the number of USS classes to an absolute minimum, you‚Äôll end up with 2-3, but any new UI will require you to make a new class. Instead if we have classes set up for individual properties, we can apply any property we want without creating new classes in the future. If a child element needs 2 of the 3 properties in scenario 1 above, you don‚Äôt need to create another USS class.

VELCRO Approach: This is how the USS would look like in VELCRO for the scenario 2 modal

.text-align-left {
  -unity-text-align: left;
}

.fs-24 {
  font-size: 24px;
}

.fs-18 {
  font-size: 18px;
}

.m-0 {
  margin: 0px;
}

.italic {
  -unity-font-style: italic;
}

.red-3 {
    background-color: var(--red-3);
}

Yes our class list is initially much larger, but now it‚Äôs fairly static over the course of the project, and standardized from one project to another. We can apply these individual classes to any element we want. Another big win here is that constantly making new USS classes is tedious. We can speed up UI development by having lots of development and pull those in when needed. Now you might ask why don‚Äôt we just use inline styles if multiple USS class performance is poor? Inline style are the worst. Inline styles are hard to debug, have the highest memory usage, can‚Äôt be re-used, stop us from implementing good practices like variables, and stop us from supporting light/dark mode. Inline styles are like using magic numbers in your code. They are evil. They are bad. They are the worst. However, that isn‚Äôt to say they should never be used. There might be small changes that are only needed once in the entire project or UI document. An acceptable location for the use of inline styles is the size property of a UI parent wrapper that is likely unique to every UXML document.

Best Practices

  1. Link the fewest number of USS classes to your UXML file as possible. The most performant solution is to put every single USS class in the entire project into 1 giant style sheet. However, that‚Äôs not the best solution when it comes to organization and development. We can ignore this performance hit and use around 2-4 style sheets per piece of UI. If the UI that is created does not require style or component classes, then those should be removed.

    1. velcro-structure.uss

    2. velcro-style.uss

    3. velcro-component.uss

    4. velcro-[UI Name].uss

  1. Use BEM (Block-Element-Modifier naming convention where possible to reduce complex USS selectors. You will find minimal complex USS selectors that rely on parent classes (Ex. .myClass1 > .myClass2 > .myClass3 OR .myClass #Element2) in VELCRO. The biggest culprits are check boxes, sliders, toggles, etc. It is necessary to use in some instances because some child elements in controls such as toggles are inaccessible.

    1. Instead of this, VELCRO uses multiple individual classes that work together and are related by class name.

      1. Ex. ‚Äú.button‚Äù (All shared USS in all button classes), ‚Äú.button-large‚Äù (All shared USS in all large button classes), ‚Äú.button-primary‚Äù (All shared USS in all button primary classes).

  1. Avoid the use of ‚Äú*‚Äù selector, It means that EVERY potential element will be tested against the selector which can be very costly as the number of elements increases.

    1. Note: The ‚Äú:hover‚Äù selector is also a ‚Äú*‚Äù selector, however this one is a necessary evil for us

  1. Try to predefine as much of your hierarchy in the UI builder ahead of time so there is as little generation at runtime as possible.

    1. Creating the UI on Awake()/Start() and hiding it with Display.None until needed is more performant than creating the hierarchy at runtime when needed and destroying it when no longer needed.

  1. All values for font, and colour should be put into USS variables in your style sheets. There should be no hard coded hex or font value in your USS classes. This stops colours/font from being changed when the theme is swapped for light/dark mode.

  1. Avoid the use of inline styles as much as possible. Inline styles override classes. it‚Äôs like an !important from CSS which makes debugging issues very difficult at times. Anything changed from the applied styles in USS classes in your properties inspector will be denoted by a little white line by the property just like when there are overrides for a prefab in the unity inspector.

Using Font in VELCRO & UI Toolkit

Font is a bit of a clown fiesta in Unity/UI Toolkit. You are unable to adjust font weights of text manually, so the best solution is to set up custom USS classes that point to a USS variable that points to a font asset that the font supports. VELCRO uses the Lato font that supports font weights of 100 (Thin), 300 (Light), 400 (Regular), 700 (Bold), and 900 (Black). However, you can still manually apply a bold or italics property in the UI toolkit inspector to an already bolded/italicized font asset. You end up with a bolded italicized bold italics font¬†.

The resulting combination is slightly different than expected and are in between the font weights/italics amount specified in the font asset.

The best solution is to set up USS variables for font in the theme files for light and dark mode. Then in velcro-style.uss, the font weight classes will point to that USS variable. The advantages to this are that DLX can override the font style if desired by creating a new theme file and pointing ‚Äú--font-bold‚Äù and others to a different font asset. Additionally this also keeps all font references out of USS classes.

When implementing font in UI toolkit, it is better to use the font weight classes in VELCRO if possible before resorting to using ‚Äú.bold‚Äù or ‚Äú.italic‚Äù classes.
