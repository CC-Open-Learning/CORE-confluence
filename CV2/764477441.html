Point & Click System


General Structure

This system relies upon the mouse’s position on the screen, and clicking in the general vicinity of the object will make the camera move to a fixed position in that direction. While in proximity / at the correct spot, the user can interact with the objects.

point and click.mp4

Mouse Placement

The mouse icon will dynamically change based on the mouse’s location on the screen. There will be a different icon for the mouse when it is hovering/pointing directly at an object that can be interacted with.

If the user can’t traverse left, the cursor will not change to point in that direction. The user has to click to interact with a particular object or move in that. direction

Movement

When standing in a space, the player will have the option to move to pre-determined positions. The user has to point in that direction and click. The player will automatically navigate to that pre-determined space. The image below shows all the possible positions in an interior space.

Based on the current location, the player can only go to these recent locations. To Turn around, they thoroughly take their mouse to the right or left side, and the player will turn around and look backwards.

If the user’s mouse faces forward, the pointer will change accordingly.

Navigation

The player [not user] when moving from one-way point to another, the player [not the user] will use navigation agents or algorithms to ensure that it avoids obstacles. We have many options at our disposal. Unity already provides some, like the nav mesh agent, Unity’s in-house navigation solution to algorithms like A* and Dijkstra’s algorithm for navigation.

Unity Nav Mesh Agent

While this in-house solution is the easiest to use, it has shortcomings. Using this tool is straightforward, but the result can sometimes leave more to desire, especially considering resource management and precision. This tool bakes a navigation area in the ground and marks obstacles when charting that course along with slope and other variables.

A*

A* is an algorithm calculated in terms of weighted graphs where the goal is to calculate the shortest route to the destination in a graphed layout. At the starting point, the algorithm will look at the G cost (distance between the end node) and H cost (distance from the end node) and add those values to make an F cost (G cost + H cost). The node will move from node to node, charting a route from start to end and calculating the F cost for each adjacent node.

Alternative Movement Options

There is a possibility that users can get dizziness from the movement in the simulation. We aim to keep the camera movement as organic as possible to avoid motion sickness. Still, in some cases, it might be inevitable, so to address that issue, we can introduce a teleportation system [Fade in / Fade out] where when the user decides to head in a particular direction will perform a fade in and out, and the camera will be at that position instead of showing the camera traverses the journey.

In this method, we don’t have to worry about navigation and can take the camera directly to its intended location. There are some con’s with this as the user might not get familiar with the layout of the surroundings, and it might be harder to envision the space around them (This con is based on a cursory glance at a potentially insignificant issue, and it could turn out not necessary as further prototypes are constructed)

Other avenues of camera controllers could also arise from this movement pattern, where a map/schematic of the environment will be present to the user, and there will be points on those maps representing the different points they can click and be teleported to.

Map Guided Navigation System

In this method, the user can view an overview of the entire map and be presented with points highlighted on the overview map. The user can click on that point and have the character move to that location. Upon reaching that position, the user can interact with items at that location. This method gives the users a view of the map, and they can easily select the destination, and their character moves to that location.

Multiple Floors / Stages

The environment could have several floors or levels that are not linked but should be easy to navigate. In that case, a menu will be made available to the users. E.g. if the sim has three floors and the user starts at floor one, and the overview map is open, then the user will be given a menu on either the top left or right with a list of the other floors.

Navigation

In this system, the map switches from a first-person perspective to an overview shot of the map. This does not have to be the entire map, as multiple floors or stages could exist. Instead of moving the camera to a different position, two cameras will use different culling masks in the rendering section, as we don’t want the user to see the movement squares in the overview section.

The player moves to that area once the user clicks on a specific point. For navigation, we can use Navmesh, as it is the in-house navigation system made by Unity. We won’t necessarily be taking advantage of the benefits of other algorithms like A*, as we have a pretty basic and static navigation workflow compared to several moving targets where the user must calculate ten if not 100’s of the object and calculate their path.

Part of the feedback received on the playtest held on the 11th of August, 2023, touched on the cons of teleporting instead of navigating to the intended destination. With Navmesh, teleporting or walking to the intended destination is straightforward.

Regular movement

player.SetDestination(hit.collider.gameObject.transform.position);

Warp / Teleport

player.Warp(hit.collider.gameObject.transform.position);

Interactions

Once the user is at their desired location, they can switch the camera to the First person person perspective and then similar to the other systems the user can select the game object they want to interact with and interact with it.
