CORE Framework : Common Pitfalls

UI Looks Completely Broken with No Styling

Possible Solution(s):

  1. Ensure the theme dropdown in the top right of the UI Builder is set to Dark.tss, Light.tss, or the DLX’s overridden theme.

Explanation: All USS variables holding colours, sprites, etc. will not display correctly if the theme file being previewed is set to Unity Default Runtime Theme. Read more at Debugging. All colours/sprites are defined in theme files. Individual USS files only have references to those variables.

Custom Controls Not Showing Up in UI Builder

Possible Solution(s):

  1. Use the template UXML file in Library Panel > Project > Assets > VELCRO UI > Templates > Toggle > Toggle.uxml

Explanation: Unlike other templates, custom controls have scripts defining their behaviour such as SlideToggle.cs and rely on the velcro-components style sheet for its visuals to be displayed. Always use the UXML templates rather than the raw scripts themselves since the templates have all the setup complete.

USS Class Properties are Not Applying to UI elements

Possible Solution(s):

  1. Unset overridden inline properties. They will always take priority over a USS class.

Explanation: An inline property is when you change a property directly on the UI element itself in the inspector rather than through a USS class. Inlined properties always have more priority than everything else meaning your classes won’t be able to overwrite those values. Inline properties are similar to !important in CSS. Inline properties are denoted by a little white bar in the UI Builder, similar to the blue bar for GameObject prefab overrides.

  1. Remove classes that target the same USS property as other properties. This is most common with margin and padding utilities.

Explanation: It is very common for UI elements to have multiple classes applied that affect the same property. The greatest USS class applied will override all other changes to that property. For example, Label elements come with a default 2px/4px margin and padding which is not used many times. Say you only want a left/right margin of 16px, an easy strategy is to first 0 everything, then apply classes p-0, m-0, ml-16, mr-16. Now say requirements change and you now require a 12px left margin. If you apply ml-12 at any point, ml-16 will still be used unless removed.

p-0 {
    padding: 0px;
}

m-0 {
    margin: 0px;
}

ml-16 {
    margin-left: 16px;
}

mr-16 {
    margin-right: 16px;
}

Hover Transitions Still Active When Element is Disabled

Hover Transitions Show Unity Default Even With :Hover States

Possible Solution(s):

  1. Ensure the “:hover” USS class also has the “:enabled” state

Explanation: Hover states will still trigger when the element is disabled if the “:enabled” state is missing in all Unity versions. In Unity 6+, the Unity Default Runtime Theme takes priority or does not trigger the “:hover" state without also defining “:enabled” or “:disabled”.

.button-primary:hover:enabled {
    border-width: 4px;
    border-color: var(--colour-button-primary-border-hover);
    background-color: var(--colour-button-primary-hover);
    color: var(--colour-text-primary-hover);
}
