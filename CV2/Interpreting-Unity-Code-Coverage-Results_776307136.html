Introduction

This page discusses interpreting your code coverage results, which sections of the HTML report you should focus on, and why.

HTML Report

When you run Unity tests with the Code Coverage package, an HTML report document is created using the results and statistics of your tests.

Summary

The first section is the Summary. This provides an overview of the test results.

Line Coverage measures how many lines have been covered by the tests that ran, throughout the entire project. While this can be a useful measurement, it’s important to employ a risk-based code coverage strategy. Be sure to focus on covering testable behaviour.

Coverage History

This section provides a line graph, which displays the general coverage history. Ideally, this graph should be level or increasing. This graph should only drop when adding new tests during development.

Risk Hotspots

To see this section in your Code Coverage summary, you must enable the Generate Additional Metrics option in the Code Coverage window.

This section displays which parts of your code have too many branches or too many paths. It’s important to know if your class is too complex or poorly covered and might need to be broken down into new classes or implementations.

The number highlighted in green generally means that the corresponding metric is acceptable, and red means it is unacceptable. However, please note that this can be followed and vary how teams see fit.

In this table, we are most concerned with Cyclomatic Complexity and CRAP score. The larger the Cyclomatic Complexity, the more complex your code is; the bigger your CRAP Score, the worse your code is. These numbers should be as low as possible, indicating clean, non-complex code. Learn more about these two topics here.

A rule of thumb for cyclomatic complexity:

  • 1-10 simple procedure, little risk

  • 11-20 More complex, moderate risk

  • 21-50 Complex, high risk

  • >50 Untestable code, very high risk

CRAP score is calculated using coverage and complexity.

Note: At the time of writing this documentation, the NPath section does not work and is not properly implemented within Unity code coverage. It will be ignored for now. This document should be updated when the NPath complexity has been properly implemented.

Coverage

This section shows individual files and the coverage statistics related to each part. Click the plus sign to display more files.

To check an individual file, double-click on the desired file to open a new page. This displays the line coverage within the code itself.

Individual File Coverage

You should see a new page like this if you double-clicked a file in the coverage section. This page shows you much more in-depth information.

Notice the information available on this page. Specifically, how the summary, coverage history and Metrics are now specific to the file you clicked on rather than a comprehensive report. In addition, you can see your code previewed with highlights in the files section.

The files section shows a preview of the code. The green code is the code that has been covered in your tests. The red lines highlight code that has not been covered. Ideally, red lines should be avoided as much as possible, as that means you have not tested this code at all.

Note: The Unity code coverage highlighting does not highlight if a line is only partially covered. For example, if you have a statement such as if(a ==5 || b == 5) and only check variable a, the highlighting will still be green even though it has been only partially covered.

Conclusion

When documenting your results, distill the necessary information (mainly the coverage), and add it to the reporting page on your project's Confluence space.

Next Steps

Now that you know how to interpret the results you were given, it is time to document them. Find out how by following the next guide on Documenting Code Coverage Results.
