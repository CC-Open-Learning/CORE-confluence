Summary

The https://bitbucket.org/VARLab/package-template repository provides a starting point for developing Unity package within the CORE Framework.

Create a Repository

With the template downloaded and ready, we need to create a new repository on Bitbucket for our package’s Unity project.

Head to the VARLab Bitbucket page https://bitbucket.org/VARLab/ and select Create → Repository.

To copy the package template repository, select Import Repository in the Create a new repository menu.

In the Import existing code menu, you will need to paste in the URL for the package template repository and provide whatever login details necessary to access it.

Your regular Atlassian password will not work for repository copying. You must generate and use an App Password for your account instead.

Download the Repository Instead

To get the template project, head over to the repository and select the … button next to Clone and select Download Repository.

You will still need to create a new repository on Bitbucket and push the downloaded package contents to that new repository.

The reason we don’t simply fork the repository is that Bitbucket offers no way to sever the fork relationship between two repositories without contacting their support staff.

Repository Settings

Ensure the repository is private, and that the main branch is named main as the scripts for publishing packages are based around that name. Other settings can be configured to the needs of the project.

Clone the Repository

Now that there’s a new repository for your package, you will need to clone it in Sourcetree or whatever git client you prefer.

For documentation on how to clone a Bitbucket repository in Sourcetree, visit Using Bitbucket & Sourcetree.

Modify the Template

Your project is now a copy of the template but we need to change around a few things in order to make your package unique. For general information on custom package creation, check out Unity’s documentation for creating custom packages.

Navigate to the Assets/Package/Runtime and Assets/Package/Editor folders and rename the VARLab.PackageTemplate assembly definition files to something that better describes your package. For example, the Cloud Save package uses VARLab.CloudSave and VARLab.CloudSave.Editor

The next step is to open package.json in the same folder and edit it to update the package namespace, display name, and description. Take a look at the sample below, and for more information check out Unity’s documentation on the package manifest.

{
	"name": "com.varlab.packagetemplate",  	<-- update namespace
	"displayName": "UPM Package Template", 	<-- update display name
	"version": "0.2.0",						<-- set a starting version, probably 0.1.0
	"unity": "2022.3",						<-- specify the minimum Unity version supported
	"description": "An example of a UPM package which can be hosted in a scoped registry.",
	"category": "Instrument",

	...

	"keywords": [							<-- add any relevant search keywords
		"unity", "editor", "upm"
	],
	
	"samples": [
		{									<-- link and describe any "sample" content
			"displayName": "Example Project",
			"description": "Contains a sample for the package",
			"path": "Samples~/Example Project"
		}
	],
	
    ...
    
}

Set Up CI/CD

The major feature of this template is that it contains scripts and pipelines that automatically create a Unity package out of anything placed in a specific folder within the project. By default the example project uses Assets/Package but you can set it to whatever you like.

To begin preparing the repository for automated packaging, head over to the Repository settings tab in your repository in Bitbucket and scroll down to Pipelines → Settings and enable pipelines.

With pipelines enabled, you can now navigate to Pipelines → Repository variables and set up the variables needed to generate a package branch.

The first variable is FOLDER_TO_EXPORT which tells the make_package.sh and publish_setup.sh scripts which folder in your Unity project contains the package files.

TARGET_BRANCH indicates the desired branch name to be used for the package. The current convention we use is upm to denote the package branch.

Most projects will not need verbose pipeline logs and can simply ignore the NPM_DEBUG variable, as it is set at the workspace level. To enable verbose debug information while running pipeline steps, add the variable and set its value to true.

Workspace-Level Variables

Variable

Secured

GIT_TOKEN

Yes

NPM_TOKEN

Yes

NPM_DEBUG

No

GIT_TOKEN is a username and app password for Bitbucket which has permissions to write to the repository.

Similarly, NPM_TOKEN is a token generated for the UPM package registry that has permissions to write to the registry.

The machine user credentials for GIT_TOKEN and NPM_TOKEN are now shared workspace variables, meaning they are accessible by all projects. These fields are Secured so that the credentials cannot be read after applying, but developers no longer need to set these variables manually for each package repository.

NPM_DEBUG is set to false at the workspace level so that individual repositories are not required to provide a value unless they need verbose debug logs.

Apply Permissions

Other developers in the VARLab will need to use your package as well, so make sure to give the Developers group write permissions for your package repository.

Navigate to Repository Settings > User and group access and add the Developers group.

If you skip this step, everyone will get errors when they open up any Unity project that is using your package.

Pushing to the Repository

With the project configured from the template, you are now ready to start developing a UPM package.

Project Version

For every feature implemented, developers should increment the package version in the package.json file, which follows semantic versioning, an x.y.z format where x is the major version, y is the minor version, and z is the patch.

See the sections below to decide which version number should be updated when making changes to the package.

Patch

If the change is particularly small, such as a simple bug fix or added function, consider bumping the patch version number. Patch-level changes should always be backward-compatible, meaning a downstream DLX project should be able to update the package without changing the use of the package.

Minor

Update the minor version when adding larger functionality, such as new classes or modules, and when fixing large bugs that affect many code areas. Minor-level changes should also always be backward-compatible, using attributes such as [Obsolete] to indicate code areas that may be removed or significantly changed in a new major version.

When the minor version number is changed, the patch version should be reset to 0

Major

For key system changes, or changes that are not backward-compatible, the major version of the package should be updated. These changes are permitted to break existing implementations in DLX projects when updated, as development teams should be aware that there may be breaking changes with new major versions.

When the major version is changed, the minor and patch versions should be reset to 0

Pipelines

Commits to any feature branch will run the Create UPM Staging Branch pipeline step which can be pulled into a project to test. A feature branch with the name of CORE-123-my-feature-branch will generate a corresponding UPM branch named upm-CORE-123-my-feature-branch

Please not that branch prefixes such as bugfix/ or feature/ are unfortunately not supported by the UPM Staging pipeline. Please avoid the use of branch prefixes on UPM package repositories. This issue is currently tracked in Jira as CORE-531.

After developing a feature and reviewing the code through a pull request, the feature branch will be merged into the main branch of the repository.

Once merged, two pipelines will run subsequently: Create UPM Branch, which pushes a commit to the upm branch of the repository (and creates it if it doesn’t already exist), and Publish UPM Package, which pushes a new version of the package to the UPM package registry. Ensure that both of these pipeline steps run successfully after merging; if a pipeline step fails, review the issues with the pipeline and contact a DNA or DevOps team member for additional support.

A successful pipeline execution means that the new version of the package will be available in the Unity Package Manager for projects that have configured the CORE UPM registry for their project.

There is a known issue with the UPM registry server currently where not all packages are visible in the Package Manager. This issue is tracked in Jira as CORE-1145 and is currently being investigated by the DevOps team.
